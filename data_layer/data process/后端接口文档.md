# 战雷遥测后端 接口文档

本文档面向前端开发，说明后端服务对外提供的 HTTP 接口、各接口返回的数据结构与字段含义。

> **契约版本：`v1.8`（2026-06-21）**。前端可据此 pin 版本；字段/枚举有破坏性变更时递增次版本号。
> v1.6：新增顶层 `replay` 标志与回放模式降级（回放时仅上报 `replay:true`，停掉全部派生数据）。
> v1.7：新增顶层 `dead` 标志与阵亡待命态（被击杀后→重生前/观战他人期间，抑制告警、置空态势/接近，战绩照常）。
> v1.8：`processed` 新增 `profile_source`/`profile_family`；机型识别加入模糊家族层（未录入的同系列变体按大类套阈值），`profile_matched` 语义扩展为“精确或家族命中均为 true”。
> 变更历史：
> - `v1.5`：新增 `awards`（战斗嘉奖：一血/双杀/三杀/连续无伤歼敌，含 `is_mine`/`notable`）与 `/api/awards`（见 4.13，实验性）；
>   修复嵌套括号载具解析（`台风 (AESA)` / `◄EF-2000 (AESA)`，惠及击杀流与嘉奖）；
>   服务(重)启/进局时排空 8111 跨局 hud 积压，杜绝上一局事件被错算（假 `you_died`）；嘉奖/掉线/连接播报从击杀流剔除。
> - `v1.4`：`combat.feed.action` 新增伤害日志动词 `lethal_hit`/`severely_damaged`/`set_afire`/`damaged`（非致命，不计 K/D）；单方 `已被摧毁`→`destroyed` 现正确计为阵亡（地面/海军 `you_died`）；掉线/连接等系统噪声从击杀流剔除。新增动作代码表。
> - `v1.3`：`indicators` 增 `weapon_firing`/`game_time`/`game_time_sec`/`gear_state`；`situation` 增 `ground_targets` 且敌我统计按单位类型过滤（`bombing_point` 不再计入敌军）；`processed.flags` 增 `overspeed_warn`/`overspeed_critical`。
> - `v1.2`：`combat` 增自我识别（`self`/`active_players`/涉我标记）、`hud_notices`、`/api/notices`、`/api/identity`。
>
> 所有字段均依据当前源码（`wt_server.py` / `wt_telemetry.py` / `wt_processor.py` / `wt_geo.py` / `wt_proximity.py` / `wt_events.py`）核实。

## 一、概述

| 项 | 值 |
|---|---|
| 服务地址 | `http://localhost:8112`（端口可用 `--port` 修改，默认 8112） |
| 数据源 | 游戏本地接口 `http://localhost:8111` |
| 响应格式 | JSON（地图图片接口除外，返回二进制图片） |
| 字符编码 | UTF-8 |
| 跨域 | 已开启 CORS：`Access-Control-Allow-Origin: *`，允许 `GET, OPTIONS` |
| 请求方式 | 全部为 `GET`（另支持 `OPTIONS` 预检） |

### 后端轮询频率（分频多线程）

后端按数据特性分四组、各自独立频率轮询游戏接口并缓存，前端按需取用即可（前端轮询频率与后端解耦）：

| 组 | 默认间隔 | 内容 |
|---|---|---|
| `fast` | 0.1s | `/state`（仪表）+ `/indicators`（座舱）+ 战局状态 |
| `map` | 0.5s | `/map_obj`（地图单位）+ 态势分析 + 接近告警 |
| `events` | 1.0s | `/mission` + `/hudmsg`（击杀/事件）+ `/gamechat`（聊天） |
| `mapimg` | 5.0s | `/map_info`（坐标换算参数）+ 小地图图片 |

各间隔可通过启动参数 `--fast-interval` / `--map-interval` / `--event-interval` / `--mapimg-interval` 调整。

## 三、完整快照 `/api/telemetry`

一次返回所有数据。结构如下（顶层字段）：

```jsonc
{
  "state": "in_battle",          // 连接状态，见枚举
  "timestamp": 1718800000.12,    // 最近一次 fast 轮询的时间戳（秒）
  "in_battle": true,             // 是否在战局
  "replay": false,               // 是否为录像回放（true 时见下方“回放模式”）
  "dead": false,                 // 是否阵亡待命（被击杀后→重生前/观战他人，见“阵亡待命态”）
  "domain": "ground",            // 当前载具域，见枚举
  "domain_label": "陆军",        // 载具域中文

  "vehicle": { ... },            // 载具仪表，见 4.1
  "indicators": { ... },         // 座舱数据，见 4.2
  "map_objects": [ ... ],        // 地图单位，见 4.3
  "map_info": { ... },           // 坐标参数，见 4.4
  "player": { ... } | null,      // 地图上“自己”那个单位（MapObject），无则 null
  "enemy_count": 7,              // 敌方单位数
  "ally_count": 5,               // 友方单位数

  "mission_status": "running" | null,
  "mission_objectives": ... | null,
  "hud_events": [ ... ],         // HUD 事件原始流，见 4.8
  "chat": [ ... ],               // 聊天，见 4.9

  "processed": { ... } | null,   // 加工数据+告警，见 4.5
  "situation": { ... } | null,   // 态势，见 4.6
  "proximity": { ... },          // 接近告警，见 4.7
  "combat": { ... } | null,      // 战绩，见 4.10
  "hud_notices": { ... } | null, // 自机技术通知，见 4.12
  "awards": { ... } | null,       // 战斗嘉奖（一血/连杀），见 4.13
  "meta": { ... }                // 各轮询组运行状态，见 4.11
}
```

### 回放模式（`replay: true`）

当玩家在游戏内观看**战斗录像回放**时，8111 仍报 `in_battle`，但数据语义与实战
完全不同：观战镜头在各载具间切换（载具/速度/油量都不属于单一玩家）、击杀会随
时间轴跳转被**重复上报**、`mission_status` 一进场就是终局结果。这类数据上报给前端
只会制造混乱，因此后端一旦识别为回放（`game_time_sec` 倒退，或进局后 `mission_status`
始终未出现 `running`），**整局降级**：所有接口（含 `/api/telemetry`、`/api/processed`、
`/api/combat`、`/api/situation`、`/api/awards`、`/api/alerts` 等）仅返回回放标记，
**不再上报告警/战绩/态势/嘉奖等任何派生数据**，直到离开战局复位。

回放时 `/api/telemetry` 形如：

```jsonc
{
  "state": "in_battle",
  "timestamp": 1718800000.12,
  "in_battle": true,
  "replay": true,
  "note": "回放模式：当前为战斗录像回放，数据语义不可靠，已暂停上报告警/战绩/态势/嘉奖等",
  "meta": { ... }
}
```

> 前端处理建议：检测到 `replay === true` 即切到“回放模式”UI（隐藏告警/战绩/态势等
> 面板，或显示“正在观看回放”），不要消费此期间任何战斗数据。`/health` 也会带 `replay` 字段，可用于轮询判断。

### 阵亡待命态（`dead: true`）

玩家被击杀后，在**可重生模式重生前**、或选择**转观战他人直到战局结束**期间，8111 的座舱遥测会
冻结在“死车残骸”上（速度 0、坠机后高度不变、乘员减员），若按活载具处理会持续刷失速/低高度/
乘员损失等**假警**；观战他人时地图“自身”坐标还会漂到被观战者身上，令态势（敌距/方位/接近）失真。

后端据此自动识别该状态（`combat.my.deaths` 增加进入；先见载具静止、再恢复运动或乘员满员退出，
即重生/再次起飞行驶），其间：

- `processed.alerts` / `processed.flags` **清空**（数值仍在，但不应据此告警）；
- `situation` 置 `null`、`proximity.events` 置 `[]`（自身位置不可靠）；
- `combat`（战绩 K/D、谁击杀了你）**照常上报**——HUD 带全局名字戳，观战期间他人的击杀**不会**被
  误算到你头上，可放心展示。

`/api/telemetry` 与 `/health` 均带 `dead` 字段。

> 前端处理建议：`dead === true` 时切到“阵亡/观战”UI，停掉飞行告警与态势面板，但可继续展示战绩、
> 谁击杀了你等信息；待 `dead` 转回 `false`（重生）再恢复。与 `replay` 相互独立。

## 四、各结构字段详解

### 4.1 `vehicle`（载具仪表，来源 `/state`）

以飞机字段为主，单位见注释。无数据时 `valid=false`、其余为 `null`。

| 字段 | 类型 | 含义 |
|---|---|---|
| `valid` | bool | 数据是否有效 |
| `altitude_m` | float\|null | 海拔高度（米） |
| `tas_kmh` | float\|null | 真空速（km/h） |
| `ias_kmh` | float\|null | 表速（km/h） |
| `mach` | float\|null | 马赫数 |
| `aoa_deg` | float\|null | 迎角（度） |
| `aos_deg` | float\|null | 侧滑角（度） |
| `load_factor` | float\|null | 过载 Ny |
| `climb_ms` | float\|null | 垂直速度（m/s） |
| `roll_rate_dps` | float\|null | 滚转角速度（度/s） |
| `fuel_kg` | float\|null | 当前燃油（kg） |
| `fuel_full_kg` | float\|null | 满油量（kg） |
| `aileron_pct` / `elevator_pct` / `rudder_pct` / `airbrake_pct` / `flaps_pct` / `gear_pct` | float\|null | 各控制面位置（%） |
| `engines` | EngineState[] | 各发动机数据，见下 |
| `raw` | object | 原始 JSON（陆战/其它字段从此取） |

**EngineState**（`engines[]` 元素）：`index`、`throttle_pct`、`power_hp`、`rpm`、`manifold_pressure_atm`、`oil_temp_c`、`thrust_kgs`、`efficiency_pct`。

### 4.3 `map_objects`（地图单位，`MapObject[]`）

坐标均为 0~1 归一化值（x 向右=东，y 向下=南）。

| 字段 | 类型 | 含义 |
|---|---|---|
| `type` | string | 大类：`aircraft` / `ground_model`（含坦克与军舰）/ `airfield` 等 |
| `icon` | string | 图标/兵种：`Player` / `Fighter` / `MediumTank` / `SPAA` 等 |
| `faction` | string | 阵营：`self` / `ally` / `enemy` / `other`（按颜色推断） |
| `color_hex` | string | 颜色 `#RRGGBB` |
| `color_rgb` | [int,int,int] | RGB |
| `blink` | int | 闪烁标志 |
| `x` / `y` | float\|null | 点坐标（归一化） |
| `dx` / `dy` | float\|null | 朝向单位向量（飞机才有） |
| `sx` / `sy` / `ex` / `ey` | float\|null | 线段端点（机场跑道等） |

### 4.4 `map_info`（坐标换算参数，来源 `/map_info`）

| 字段 | 类型 | 含义 |
|---|---|---|
| `valid` | bool | 是否有效 |
| `map_generation` | int\|null | 地图版本号（换图递增） |
| `grid_size` | [float,float]\|null | 网格参考区域尺寸（**注意：不是整图尺寸**） |
| `grid_steps` | [float,float]\|null | 网格格距（米） |
| `grid_zero` | [float,float]\|null | 网格原点（米） |
| `map_min` / `map_max` | [float,float]\|null | 地图世界边界（米）。**真实距离换算用 `map_max - map_min`** |
| `raw` | object | 原始 JSON |

### 4.5 `processed`（加工数据 + 告警，核心）

后端基于机型配置（`vehicle_profiles.json`）计算的派生信息与告警。

> **机型识别三级（模糊家族识别）**：收到 `vehicle_type` 后依次尝试——①在配置表精确命中
> （`profile_source=exact`）；②未命中且为空军时，按“厂商+型号”家族前缀做模糊识别
> （`profile_source=family`，例如 `fw-190a-9`、`bf-109g-14` 等未单独录入的亚型会归到 `Fw 190`/`Bf 109`
> 家族，套用该家族大类阈值）；③仍不中才回退默认值（`profile_source=default`）。同系列变体的失速/超速/
> 高度等阈值差异不大，按家族大类处理即可。前端若需展示“识别到的机型/家族”，`profile_source=family`
> 时用 `profile_family`，`exact` 时用 `vehicle_type`。

| 字段 | 类型 | 含义 |
|---|---|---|
| `vehicle_type` | string\|null | 载具代号 |
| `profile_matched` | bool | 是否套用到了机型适配阈值（精确或家族模糊命中均为 `true`；`false`=回退默认值） |
| `profile_source` | string | 阈值来源：`exact`=精确命中机型 / `family`=模糊家族命中（同系列变体按大类套用） / `default`=回退默认 |
| `profile_family` | string\|null | 模糊命中的家族标签（如 `"Fw 190"`、`"F-4 Phantom II"`），仅 `profile_source=family` 时有值 |
| `army` | string\|null | 兵种 |
| `vehicle_class` | string | 归类：`air` / `ground` / `heli` / `unknown` |
| `fuel_fraction` | float\|null | 剩余燃油比例 0~1 |
| `fuel_kg` | float\|null | 当前燃油 |
| `fuel_burn_rate_kgs` | float\|null | 估算油耗（kg/s） |
| `fuel_remaining_sec` | float\|null | 估算剩余可飞时间（秒） |
| `afterburner_active` | bool | 是否开加力 |
| `afterburner_elapsed_sec` | float | 连续加力时长（秒） |
| `afterburner_max_sec` | float\|null | 加力时间上限（0/null=不限） |
| `ias_kmh` / `aoa_deg` / `altitude_m` | float\|null | 表速/迎角/高度 |
| `g_now` / `g_max` / `g_min` | float\|null | 当前/本架次峰值/谷值过载 |
| `crew_current` / `crew_total` | float\|null | 坦克乘员存活/总数 |
| `ammo_first_stage` | float\|null | 一级待发弹 |
| `gun_stabilizer` | bool\|null | 火炮稳定器 |

| 字段 | 类型 | 含义 |
|---|---|---|
| `id` | int | 事件 id |
| `time` | int\|null | 游戏内时间 |
| `action` | string | 动作代码（见下表） |

| `parsed` | bool | 是否成功解析 |
| `raw` | string | 清洗后的原文 |

**动作代码（`action`）**：分两类，`is_kill` 标明是否计入 K/D。

| 代码 | 标签 | `is_kill` | 来源/实测措辞 |
|---|---|---|---|
| `shot_down` | 击落 | ✓ | 击杀榜：`X 击落了 Y`（空战互杀） |
| `destroyed` | 击毁 | ✓ | 击杀榜：`X 击毁了 Y` / 单方 `X 已被摧毁`（地面/海军阵亡） |
| `crashed` | 坠毁 | ✓ | 单方：`X 已坠毁。`（飞机坠机/被击落后坠地） |
| `wrecked` | 损毁 | ✓ | 单方：`X 已失控` |
| `lethal_hit` | 致命一击 | ✗ | 伤害日志：`X 致命攻击 Y`（细粒度，常打 AI bot，**不计 K/D** 以免与击杀榜重复） |
| `severely_damaged` | 重创 | ✗ | 伤害日志：`X 重创 Y` |
| `set_afire` | 点燃 | ✗ | 伤害日志：`X 的攻击引燃了 Y` |
| `damaged` | 击伤 | ✗ | 伤害日志：`X 击伤 Y` |

> 同属 `kind=='damage'` 的两类文本在一局里并存：**击杀榜**（致命，权威 K/D）与**伤害日志**（细粒度，非致命）。
> 系统噪声（`X 已掉线。` / `NET_PLAYER_DISCONNECT_FROM_GAME` / 连接/加入）已在数据层剔除，不会出现在 `feed`。

> **击杀流的敌我标注**：游戏 API 不暴露其他玩家的队伍（hudmsg `enemy` 字段实测恒为 false），
> 无法可靠给击杀流每个名字标敌/友；故仅在识别出“自己”后标注 `involves_me/is_my_kill/is_my_death`。
> 地图单位的敌我识别请用 `map_objects[].faction` / `situation`（按队伍颜色，可靠）。

#### 事件消费建议（`you_killed` / `you_died`）

> 针对插件侧 `you_killed`/`you_died` 事件，数据层给出的可靠消费方式（基于真机抓包验证）：
>
> - **`you_killed`**：监听 `feed[]` 中 `is_my_kill == true` 的**新 id**（按 id 去重；多杀可在窗口内合并）。
>   击杀文本如 `我 (载具) 击毁了 敌`。
> - **`you_died`**：监听 `feed[]` 中 `is_my_death == true` 的**新 id**。**这是唯一可靠的死亡信号**。
>   实测覆盖三种措辞，均正确触发 `is_my_death`：
>   - 被真人击落（双方形式）：`敌 (苏-30SM2) 击落了 我 (歼-15T)` —— `action=shot_down`；
>   - 自身坠机（单方形式）：`我 (歼-15T) 已坠毁。` —— `action=crashed`；
>   - 地面/海军被摧毁（单方形式）：`我 (哈立德 I) 已被摧毁` —— `action=destroyed`。
>   注意被击落前常先有一条伤害日志 `敌 致命攻击 我 (...)`（`action=lethal_hit`，非致命，**不触发** `is_my_death`），
>   真正的死亡以随后的 `击落了`/`已坠毁`/`已被摧毁` 为准。
> - **不要用 `valid` 翻转判死亡**：实测整局（1181 帧）`/api/state.valid` 与 `/api/indicators.valid`
>   **全程为 true，从不翻转**，无法用于判定阵亡/退场；`mission_status` 仅用于战斗结束（`battle_end`）。
> - 前提：`you_killed`/`you_died` 都依赖已确定“自己”（`combat.self.name` 非 null）。未识别出自己时这两个标记恒为 false，
>   请先经 `/api/identity` 设定或等待 `auto` 置信度达标。

### 4.12 `hud_notices`（自机技术通知，来源 `/hudmsg` 文本解析）

### 4.13 `awards`（战斗嘉奖，来源 `/hudmsg` 文本解析）｜实验性

从 HUD 文本流解析出的**战斗嘉奖/连杀播报**，天然带玩家名+战队+载具，可用于做高光提示，
或识别敌方高威胁玩家（连杀王）。实测格式：
`<[战队] 玩家名> (载具) 获得嘉奖“双杀！”`，一血为独立句式 `<玩家名> (载具) 先拔头筹！`。

> 与 `combat`（击杀流）区分：嘉奖无“击杀者→被击杀者”结构，已从击杀流剔除、单独成流。
> 按 `(kind, id)` 去重，新出现一条推一条。对应接口 `GET /api/awards`（`?notable=1` 仅返回高光子集）。

| 字段 | 类型 | 含义 |
|---|---|---|
| `total` | int | 本局累计嘉奖数 |
| `by_code` | object | 各嘉奖代码 -> 次数 |
| `feed` | Award[] | 全部已识别嘉奖（最新在前，上限 60） |
| `notable` | Award[] | **罕见而重要**的子集（一血/连杀/连续无伤歼敌等），建议前端只展示/推送这个 |

**Award**（`feed[]` / `notable[]` 元素）：

| 字段 | 类型 | 含义 |
|---|---|---|
| `id` | int | 来源 HUD 消息 id |
| `time` | int\|null | 游戏内时间 |
| `player` / `squad` / `vehicle` | string | 获奖者 名/战队/载具 |
| `is_ai` | bool | 是否 AI |
| `code` | string | 嘉奖代码，见下表（未知嘉奖为 `award_other`） |
| `label` | string | 嘉奖原始名称（如 `连续无伤歼敌×3`） |
| `tier` | string | `major`（高光：一血/连杀）/ `notable` / `minor` |
| `notable` | bool | 是否罕见重要（决定是否进 `notable` 子集） |
| `is_mine` | bool | 是否“自己”获得（依赖已识别 `combat.self`） |

## 七、重要说明与已知局限

1. **敌人型号不可知**：游戏只暴露敌人的**兵种类别**（`icon` / `category`），不提供具体型号；具体型号仅自己的载具有（`indicators.vehicle_type`）。
2. **坦克与军舰同为 `ground_model`**：`type` 无法区分二者，需结合 `domain` 判断。
3. **海军无仪表数据**：海军模式下 `/indicators`、`/state` 多为无效，`vehicle`/`indicators` 字段大多为 `null`，但地图/态势/接近告警仍可用。
4. **距离换算**：基于 `map_max - map_min`；若地图无该参数则回退 `grid_size`（可能不准）。
5. **对战类型不可知**：游戏接口不提供街机/真实/全真（AB/RB/SB）信息。
6. **温度告警按发动机类型自动分流**：液冷看水温、气冷看油温+缸头、喷气看涡轮温度；不适用者字段为 `null` 且不告警。
