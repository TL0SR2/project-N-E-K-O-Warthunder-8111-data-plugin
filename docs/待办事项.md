# 待办事项

> 本文件给**数据层开发者**看：列出 Battle Awareness（猫娘副驾驶）侧在 **DTO 冻结前**需要数据层补齐的事项。
> 两侧唯一边界 = HTTP `:8112`（`/api/telemetry`）。以下都是"在你现有架构上加字段/补解析"，不改你的结构。

## 优先（DTO 冻结前必须）

- [ ] **T-Safety: output text sanitizer**
  - 目标：防止猫娘复读不良玩家 ID、`hudmsg`、`combat.feed` 原文，避免辱骂、涉政、擦边、仇恨、广告、联系方式、奇怪符号或 prompt injection 文本进入猫娘输出。
  - 放置位置：`neko_warthunder` 的 `NekoDispatcher` / prompt builder 前。数据层继续保留 raw fact，不负责输出安全；Detector / Scenario / Arbiter 不承担文本过滤职责。
  - 关键策略：
    - raw 玩家名 / hudmsg / combat.feed 原文只进 audit/debug。
    - safe 字段才允许进入 prompt。
    - 默认 generic 文案，不朗读陌生玩家名（例如"一名敌人"、"对手"、"某位玩家"）。
    - `hudmsg` / `combat.feed` 原文默认不进 prompt。
    - 不确定时宁可不读原文。
  - 阻塞关系：阻塞 kill / death / hudmsg / combat.feed 正式播报；不阻塞 `stall_risk` / `low_alt_danger` / `overheat` / `low_fuel` 等数值安全事件。
  - 实现前测试：sanitizer 单测；dispatcher prompt 测试；integration 测试；`push_message.parts[].text` 不包含 unsafe raw 的合同测试；dry_run 能显示 `redacted_reason` / `text_safety_level`。

- [ ] **补 overspeed（超速）告警 flag**
  - 现状：`/api/processed` 已有 stall / aoa / altitude / fuel / overheat 等告警，但**没有超速告警**；`vehicle_profiles.json` 也没有 never-exceed（最大安全速度）字段。
  - 需要：
    - 在 `vehicle_profiles.json` 加逐机/逐类 never-exceed 阈值（建议 `overspeed_warn_kmh` / `overspeed_critical_kmh`；按 IAS 还是 Mach 由你采样定）。
    - 在 `wt_processor.py` 加 `overspeed_warn` / `overspeed_critical` 两级告警 code（可参照 `_process_stall` 写法）。
  - 目标：`/api/processed.flags` 出现 `overspeed_warn` / `overspeed_critical`，我们消费其边沿。

- [ ] **完成 hudmsg 事件解析与推送**（与你现有待办一致）
  - 现状：hudmsg（击杀 / 起火 / 重创 / 油温 / 非对称襟翼）解析尚未完善，`hud_events` / `combat.feed` 还不稳定。
  - 需要可靠解析为结构化、带**递增 id** 的事件，至少包含：
    - 击杀（击落 / 击毁）
    - 引燃（着火）
    - 重创敌人
  - 目标：我们的 `you_killed` / `you_died` 能稳定从 `combat.feed` / `hud_events` 按 id 去重消费。

- [ ] **击杀归属判断（is_kill + 玩家本人）**
  - 现状：`combat.feed` 已有 `is_kill` / `killer` / `victim` / `killer_is_ai`，但"这条是不是玩家**本人**击杀/被击落"取决于玩家名匹配。
  - 需要：明确 `killer == player_name`（我击杀）与 `victim == player_name`（我阵亡）的判定，并区分助攻与 AI（`killer_is_ai`）。
  - 目标：我们能区分"我击杀 / 我被击落 / 队友阵亡"三种语义。

- [ ] **player_name 注入方式确认**
  - 现状：`combat.player_name` 需启动参数 `--player-name`；运行时如何稳定拿到玩家本人昵称尚未约定（8111 不直接暴露玩家本人名，是已知难点）。
  - 需要：确认注入方式（启动参数 / 配置文件 / 能否从其它途径自动取）。
  - 目标：稳定拿到 player_name，作为上面"击杀归属判断"的依据。
